# 设计哲学

## 为什么需要对抗式协作？

AI 辅助编程的核心困境：**单个 AI（或单个 prompt）存在系统性认知盲区**。它可能：

- 忽略边缘场景和极限输入
- 默认使用过时或错误的外部知识
- 在安全/合规问题上缺乏质疑
- 为"完成任务"而牺牲质量

Swarm/Legion 的解法：**强制多角色隔离审查**，让不同的 Agent 子代理以不同的视角审视同一任务。

---

## 核心原则

### 1. 对抗产生质量

R1（需求撕裂者）的任务就是**找茬** — 发现需求中的矛盾、模糊和遗漏。R2 负责**事实核对** — 验证每个技术假设。R4 独立测试，不看实现细节，只对照契约验收。

### 2. 隔离确保独立

R1/R2/R3/R4 必须通过 **Agent 子代理** 真隔离执行，禁止主对话分饰多角。这确保每个角色的判断不受其他角色影响。

### 3. 闸门保证秩序

```
R1 + R2 并行 → 两者关门 → R3
R4a ∥ R3（黑盒，契约级）
R3 关门 + 证据回灌 → R4b
统帅按 R4a 表亲测 → PASS / COND_PASS / FAIL
```

### 4. 证据分级 (A/B/C)

| 等级 | 含义 | 可否写入已拍板 |
|------|------|--------------|
| **A（可验证）** | 附带可复核证据：URL+条款、命令+退出码+输出、路径+行号 | 是 |
| **B（待核实）** | 逻辑自洽但未对外复核 | 否 |
| **C（观点）** | 偏好、经验法则、无证据链 | 否 |

### 5. 可缩放治理

- **L1**：日常小任务，单人直执行
- **L2**：有实质风险的代码改动，蜂群 4+1
- **L3**：大型多模块项目，军团 EX/MG/OP + 末端蜂群

---

## 与其他方法论的区别

| 方面 | 传统 Code Review | Swarm/Legion |
|------|-----------------|--------------|
| 审查时机 | 代码写完之后 | 写码前（R1/R2）→ 写码中（R3）→ 写码后（R4）全流程 |
| 审查维度 | 代码风格+逻辑 | 需求完整性 + 技术事实 + 安全合规 + 契约验收 |
| 执行方式 | 人工 | AI Agent 子代理自动化 |
| 证据标准 | 无强制 | A/B/C 三级 + 裁决 PASS/FAIL |

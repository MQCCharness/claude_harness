# 军团模式：军令状与信息穿透机制

**范围**：军团模式阶段四（军令状生成）的操作细则，以及全流程中信息传递的穿透纪律。

## 1. 军令状模板（可复制）

```markdown
# 军令状（Mission Brief）

**任务链 ID**：___
**生成时间**：___
**军团统帅**：主对话助手

---

## ① 原始需求（用户原话，禁止改写）

> [此处粘贴用户原始消息全文]

## ② 冻结目标句（统帅提炼，用户已确认）

[一句话目标]

## ③ 硬约束（不可违反）

- [PENETRATING] <安全/合规/数据隐私等关键约束>
- <技术限制>
- <工期/资源约束>
- …

## ④ 非目标（明确不做）

- <非目标 1>
- <非目标 2>
- <非目标 3>
- …

## ⑤ 成功标准（可验证）

- <标准 1：可测条件>
- <标准 2：可测条件>
- …

---

🔒 本文档为军令状，任何层级 Sub-agent 不得修改。发现矛盾须上报军团统帅。
```

### 1.1 执行大环境（附录 · 与渐进披露 / WAVE 接力配套）

**目的**：规定**产出与操作的大环境基础**（仓库根、关键路径、ENV、容器等）。**禁止仅写在聊天**；Summarize 与记忆接力只 **Read 落盘**，无法恢复从未写入军令状 / HANDOFF / `CHAIN-STATE` 的口头约定。

**宿主泛化条文**：`.claude/swarm/PORTING-PROMPT-PROGRESSIVE-MEMORY-ASK.zh-CN.md` §2.1。

**推荐**在军令状正文末或单独附录增加下表（**变量只写名与指针**，密钥取值不进仓）：

```markdown
## ⑥ 执行大环境（附录）

| 项 | 值或指针（相对仓库根路径优先） |
|----|--------------------------------|
| 仓库根 / monorepo 子包 | 例：`./` 或 `./packages/foo` |
| 关键路径 | 例：构建入口、配置目录 |
| 环境变量 | **名** + 指针 → `.env.example` 或 CI 文档 |
| 容器 / 运行时 | 例：compose 服务名、镜像 tag、devcontainer 路径 |
| 跨 WAVE 不变量 | 仍适用的约定或「见 WAVE-CLOSE-… §x」 |
```

**纪律**：阶段四冻结军令状时统帅**核对**本附录是否足以让下一会话 / 下一 **WAVE** 在**不依赖聊天全文**下复现操作面；缺口则补写后再冻结。多波次收口时若大环境变更，须同步修订附录或 **HANDOFF** 写清 delta。

## 2. 穿透规则详解

### 2.1 军令状传递纪律（按三层级细化）

| 传递路径 | 发送方 | 接收方 | 穿透规则 | 可追加内容 |
|---------|--------|--------|---------|-----------|
| 统帅 → EX | +1 | EX | 军令状**全文或落盘路径**（必须） | 战略指令区 |
| EX → MG（经统帅路由 **或** EX《下级 Task 发包》+ 统帅代发） | EX | MG | 军令状**全文或落盘路径**（必须） | 战略约束（标注追加者+层级=EX）；发包须满足 `CLAUDE.md §三` 中间层分发义务 |
| MG → OP（经统帅路由 **或** MG《下级 Task 发包》+ 统帅代发） | MG | OP | 军令状**全文或落盘路径**（必须） | 执行约束（标注追加者+层级=MG）；发包同上 |
| OP → 蜂群 R1–R4 | OP | L2 | **至少**携带军令状 ②③⑤ | 蜂群统帅按`CLAUDE.md §二` 裁量 |
| 任何层级发现矛盾 | — | — | **上报**直接上级，最终达军团统帅 | — |

**注（与 `CLAUDE.md §三` 对齐）**：**L3 角色**（EX / MG / OP，含叶 OP）各对应一条军团架构上的 Task，穿透区须军令状**全文或落盘路径**。**R1–R4** 为 **L2** 蜂群内角色，**不**单独占用军团架构节点计数；由下表「OP→蜂群」行约束**至少 ②③⑤**，且 §2.5 **`[PENETRATING]`** 原文全链不变，常与叶 OP 穿透区中的路径 + Read 衔接。

### 2.2 EX 层穿透规则

- **可追加**：战略约束（技术路线偏好、跨域优先级排序等）
- **追加标注**（强制）：`追加者=<EX角色名> 追加层级=EX 追加轮次=ROUND-<N>`
- **追加效力**：**软约束**——下级应遵循，但与军令状硬约束冲突时以军令状为准
- **不得**：修改军令状五要素；替下级做实现决策

### 2.3 MG 层穿透规则

- **可追加**：执行约束（资源分配、时序依赖、工程规范等）
- **追加标注**（强制）：`追加者=<MG角色名> 追加层级=MG 追加轮次=ROUND-<N>`
- **追加效力**：**执行级约束**——OP 须遵循，但与军令状或 EX 战略约束冲突时以上级为准
- **不得**：修改军令状或 EX 追加的战略约束；跳过 OP 直接产出实现方案

### 2.4 冲突裁决序

军令状五要素（最高）> `[PENETRATING]` 标签 > EX 战略约束 > MG 执行约束 > OP 本层补充（最低）。冲突须**上报**而非自行裁决。

### 2.5 `[PENETRATING]` 标签

军令状 ③ 硬约束中标记了 `[PENETRATING]` 的条目：

- **传递时必须原文复制**，贯穿 EX → MG → OP → 蜂群全链路，不得摘要、改写或省略
- **优先级最高**：与任何层级追加信息冲突时，以 `[PENETRATING]` 为准
- **冲突上报**：任何层级发现冲突时须在向上汇流中**显式标明**冲突条目与层级
- **典型使用场景**：安全底线、法规合规、数据隐私、密钥策略、多租户隔离

### 2.6 禁止行为

- 禁止任何层级**删除**军令状中的条目
- 禁止将 EX/MG 的**推断**混入军令状正文（须放在追加区并标注追加者与层级）
- 禁止以「摘要」为名**替换**军令状原文（摘要仅用于上游摘要链，军令状本身不被摘要）
- 禁止假设下级「已经知道」军令状内容而不传递
- 禁止 MG 修改 EX 的追加内容（可追加新条目，不可改已有）
- 禁止追加内容**不标注**追加者与层级（无标注视为违规，Gate 时须拦截）

### 2.7 军令状冻结确认、修订与 G/L/O 标注（操作细则）

- **冻结前确认**：军令状 **②③④⑤** 定稿前须具备 **用户明示**（原话或可指针对象），或 **统帅草案 + 用户确认**（双留痕）；与 `CLAUDE.md §三` **军令状修订与穿透区正文的纪律**一致。**+1 禁止**在无上述确认时单独改写穿透区五要素或弱化 `[PENETRATING]`。
- **矛盾分流**：事实核对发现契约与证据不一致时，由统帅归类 **G（全局）/ L（局部）/ O（操作）**（定义与动作表见`CLAUDE.md §三` **事实核对矛盾 · 多路升级**）；Sub-agent 仅输出**建议分类 + 证据等级**，**不得**自断全局。
- **EX/MG 追加区**：凡外部事实、厂商行为、可达性等 **R2 域**陈述，默认 **B/C** 或附 **待 R2**；**禁止**标 **A** 除非指回 **R2 Task 关门**或 **统帅可复核指针**。
- **`decisions_log`（推荐）**：军团层新增矛盾处置条目时建议带 `SCOPE=G|L|O`；若 `AFFECTS_SHAPE=yes` 而未获用户明示推翻与统帅留痕对齐形状，则 **Gate 不得假放行**（与`CLAUDE.md §三` 同条）。

**可复制附言（贴入 `MISSION-BRIEF` 文末可选）**：

```markdown
## 冻结确认（示例）
- [ ] ②③④⑤ 已由用户明示确认 **或** 草案已获用户确认（指针：___）
- [ ] 已知矛盾分流将适用 G/L/O（见 `CLAUDE.md §三`）
```

## 3. 上下文摘要契约（详细说明）

### 3.1 三区结构

每层向下传递的 Task 正文包含三个语义区：

**穿透区**（不可改）：
- 军令状全文（首选）
- 或军令状落盘路径 + `请 Read 此路径获取军令状`

**本层追加区**（该层角色写）：
- **本层理解**：一句话说明本层角色如何理解自己要做的事
- **领域知识补充**：本层专业领域内对下级有帮助的信息
- **交付物定义**：明确期望下级产出什么、格式、粒度
- **时序或依赖说明**：下级产出是否依赖其他部门结果

**I/O 路由输入段**（Gate 通过后、统帅为下游 Task 组装时使用）：

统帅在「本层追加区」中追加以下结构，将上游产出路由给下游角色：

```
[I/O 路由输入]
依据 I/O 契约表，你的输入如下：

| 来源角色 | 产出名称 | 内容（摘要或落盘路径） |
|---------|---------|-------------------|
| 角色A | 产出X | （粘贴关键内容或路径） |
| 角色B | 产出Y | （粘贴关键内容或路径） |

你的输出须符合 I/O 契约表中的「输出格式规范」：<此处复制该角色的格式规范>
你的输出须包含 decisions_log（已决策 + 排除项 + 理由）。
```

**I/O 路由规则**：
- 内容短（<500 字）→ 直接粘贴到 Task 正文
- 内容长 → 提供落盘路径 + 关键段落摘要
- **禁止**将上游全部原始产出不裁剪地一锅端
- **禁止**将 I/O 路由段混入穿透区

**禁区**（强制遵守）：
- 不得引用**同层级他部门**的结论，除非军团统帅已按 I/O 契约裁剪转发
- 不得将**推断**标为「已确认」
- 不得把本层理解**混入**穿透区军令状正文

### 3.2 为何这样设计

| 区域 | 解决的问题 |
|------|-----------|
| 穿透区 | 防止原始需求在多层传递中**丢失** |
| 本层追加区 | 允许每层加入专业知识（分层的**价值所在**） |
| 禁区 | 防止中间层的理解**污染**原始约束 |

### 3.3 逐级主题汇流块（向上汇报）

各 L3 角色向上级（及 OP 向 MG/统帅）交付时，须含 **《逐级主题汇流块》** 表格字段：本层职责一句、直连下级清单与状态、下级到齐状态、主题摘要、矛盾与风险、上送指针。字段语义与 OP/蜂群收口例外见 **`CLAUDE.md §三`** 专节。推荐可另落盘 `legion/rollups/ROLLUP-<CHAIN>-<TIER>-<slug>.md`。

### 3.4 小组信息空间（与穿透并列、不替代）

**领导辖内**执行期共识区：路径与边界见 **`CLAUDE.md §三`「小组信息空间」** 与 **`legion/group-space/README.md`**。组空间**不得**升格为穿透区或替代军令状；跨组引用须经统帅 I/O 裁剪。

## 5. 常见误读

| 误读 | 正读 |
|------|------|
| 「军令状太长，下级不看」 | 军令状是**穿透文档**，每层 Task **必须**携带；不读不是不传 |
| 「中间层可以替下级做决定」 | 中间层可追加知识，但**不得**替代军令状的硬约束 |
| 「蜂群 R1–R4 也要看完整军令状」 | 蜂群 R1–R4 **至少**看 ②③⑤；完整军令状由蜂群统帅裁量 |
| 「穿透 = 所有信息都传下去」 | 穿透仅指军令状五要素；其他信息仍按最小充分原则裁剪 |
| 「只有统帅能下发 Task」 | **带下级 L3 角色**（含架构许可的非叶 OP）可起草**直接下级**《下级 Task 发包》；平台限制时由 **统帅代发**（见 `CLAUDE.md §三`） |
| 「组空间 = 全局事实」 | 组空间仅**组内执行期**有效；**不得**替代 ROLLUP 必填项与军令状 |
| 「`CLAUDE.md §三` 说每个 Sub-agent 都要军令状全文」 | **L3 架构角色 Task**须全文或路径；**R1–R4** 按本章 **至少 ②③⑤** 与 §2.5 **`[PENETRATING]`**，并遵`CLAUDE.md §二`；二者**分层衔接**，不是两套互相取消的义务 |
| 「+1 可以顺手改军令状省事」 | 穿透区修订须 **用户明示** 或 **草案+用户确认**；见 §2.7 与`CLAUDE.md §三` |
| 「EX/MG 写『已核实 A』就算 R2」 | **R2 型 A** 仅 R2 关门或统帅可复核指针；EX/MG 默认 B/C |

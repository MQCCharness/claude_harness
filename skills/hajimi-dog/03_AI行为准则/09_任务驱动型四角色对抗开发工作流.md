# 任务驱动型对抗开发工作流 (Task-Driven Adversarial Workflow)

为了在具体的软件开发或复杂任务中最大化提高交付质量，我们将原本用于系统自我进化的"四角色混沌对抗架构"重塑为一套**高标准的工业级软件开发协作流水线**。

该架构的核心理念是：**拒绝"能跑就行"的顺向思维，通过内部博弈、极端场景拷问和外部事实校验，倒逼代码质量的提升。**

在深度演进中，我们引入了 **"4+1" 架构（四执行角色 + 一统帅）** 以及 **Sub-agent 物理隔离机制**，这不仅解决了大模型长上下文带来的幻觉问题，更与当前业界最顶尖的多智能体（Multi-Agent）工程化思想不谋而合。

**与 Claude Code 工作区硬规则对齐（必读）**：「4+1」指统帅总线纪律；**不等于**每条用户消息都自动四轮 Agent 子代理。**何时必须 Agent 子代理**、**满编口令**、**豁免关键词**、**规范栈优先级** 以 CLAUDE.md §二 为准（见该文件「4+1 默认形态与 Agent 子代理强制边界」「**统帅实现轨外置**」「形状保持与防目标漂变」「规范栈、优先级与冲突裁决」）。**用户话术 → 建议 Read 的路由表、续聊状态模板、极端情况备忘**见 `.claude/skills/hajimi-dog/03_AI行为准则\16_蜂群触发路由与形状保持.md`。维护模板与审计清单见 `.claude/skills/hajimi-dog/03_AI行为准则\12_蜂群规则与技能体系维护指南.md`。泛化一页摘要：`.claude/swarm/统帅只负责调度.md`。

## 架构全景："4+1" 角色分工与职责

### 0. 统帅 (Orchestrator / PM) —— 负责"目标锚定与全局调度"
* **核心目标**：防止对抗失控，确保所有角色的博弈最终收敛于用户的"原始需求"，避免过度设计。
* **具体任务**：
  * **目标锚定 (Goal Alignment)**：死死攥住用户的原始需求。每当四角色完成一轮博弈，统帅负责校验："当前讨论的极端防御方案，是否偏离了核心业务目标？"
  * **打破死锁 (Deadlock Resolution)**：当撕裂者提出极低概率的物理级灾难，导致开发者与测试员陷入无限报错循环时，统帅负责拍板："记录为已知风险，降级处理，强制推进主流程。"
  * **上下文压缩 (Context Summarization)**：在每个阶段结束时，把长篇大论的争吵压缩成"核心结论"传递给下一环节，极大节省 Token 并防止上下文溢出。
  * **R4 清单执行门禁与验收裁决**：仓库改动**落地后**（**默认**由 **R3 Agent 子代理** 落笔，见 CLAUDE.md **统帅实现轨外置**；用户明示 **`主帅直接改仓`/`统帅可落地代码`** 时主帅直改须首段声明），按 R4 产出的清单**实测**（测试/起服务/关键路径），给出 **PASS / COND_PASS / FAIL**；**P0 未闭环不得收束**；必要时**再次满编 R1–R4**直至满足或用户明示豁免（与 Claude Code 工作区 CLAUDE.md §二 对齐）。
  * **领域不熟时的需求升格**：当用户不熟细分领域、初版需求缺少行业基准时，**优先**按技能 `.claude/skills/hajimi-dog/03_AI行为准则\11_领域不熟与需求升格蜂群协议.md` 执行 **Phase A（R2 对标）→ Phase B（MVP/标准/野心 + R3 性价比）→（可选）R1 竞争力攻击**，由统帅冻结契约后再进入常规开发与写码回合；规则层见同工作区 CLAUDE.md §二「领域不熟与需求升格」。
  * **R1/R2 转化与反孤岛**：禁止只「朗读」R1/R2 子代理长文；每条结论须落到 **→R3 约束 / →R4 条目 / →已知风险 / →待用户决策** 之一；R2 事实须写成 **R3 可执行约束**；R1/R2 晚于 R3/R4 暴露问题时须 **二次分发** Agent 子代理（见同规则 `R1/R2 产出转化义务`）。
  * **第三方 API / LLM 集成**：对接外部网关时先做 **问题分层**（传输/契约/策略/解析），**R2 事实与 curl 复现** 优先于多轮盲改解析；**R4** 不得仅以 mock 充当 LLM 路径 **全部** P0；子代理「已实现」**须**经统帅 **diff+test** 核验。细则见 `.claude/skills/hajimi-dog/03_AI行为准则\13_第三方API与集成层蜂群协议.md` 与同工作区 CLAUDE.md §二「第三方 API、LLM 与集成层」。
* **存在价值**：作为整套体系的"方向盘"，确保所有的混沌对抗都在为"交付用户想要的结果"而服务。

### 1. 需求撕裂者 (Requirement Tearer) —— 负责"破坏与发难"
* **前身角色**：改变规则的出题人 (Chaos Examiner)
* **核心目标**：在写下第一行代码前，打破"理想状态"的幻想，找出所有可能导致系统崩溃的边缘场景（Edge Cases）。
* **具体任务**：
  * **极限施压**：提出极端网络、极端硬件条件下的系统表现问题（如："断网/断电怎么办？"）。
  * **性能拷问**：针对并发、内存溢出、超大数据量的极端输入进行质询。
  * **恶意输入**：模拟黑客或小白用户的非法输入（如：SQL注入、超长乱码）。
* **存在价值**：逼迫架构设计在初期就必须考虑高容错性、异常捕获和降级策略。

### 2. 技术调研员 (Technical Researcher) —— 负责"开眼看世界"
* **前身角色**：外部审计员 (External Auditor)
* **核心目标**：打破 AI 的信息茧房和代码幻觉，引入真实世界的最佳实践和最新官方文档。
* **具体任务**：
  * **技术选型校验**：面临不熟悉的 API 或底层机制时，强制联网查阅官方文档（如：PyTorch 官方规范、最新前端框架特性）。
  * **排雷与避坑**：检索 GitHub Issues 或 StackOverflow，确认第三方库是否存在已知的致命 Bug。
  * **打破死胡同**：当开发陷入"玄学"报错时，强制介入寻找外部真实世界的解决方案。
* **存在价值**：确保技术方案的绝对正确性和前瞻性，避免闭门造车。

### 3. 核心开发者 (Core Developer) —— 负责"建设与防御"
* **前身角色**：体系优化者 (System Optimizer)
* **核心目标**：在"需求撕裂者"的压力和"技术调研员"的蓝图下，编写健壮、优雅、可维护的业务代码。
* **具体任务**：
  * **防御性编程**：将"需求撕裂者"提出的所有极端场景转化为代码中的 `try-catch`、断言（Assert）、超时处理和状态机回滚逻辑。
  * **核心逻辑实现**：遵循 SOLID 原则，编写高内聚、低耦合的模块化代码。
  * **规范落地**：严格遵守项目的 Lint 规则和代码风格。
* **存在价值**：将对抗产生的理论要求，转化为坚如磐石的实体代码。

### 4. 独立测试员 (Independent Tester) —— 负责"无情诊断"
* **前身角色**：QA 检测者 (QA Tester)
* **核心目标**：站在用户的角度，对"核心开发者"交付的代码进行冷酷无情的暴力测试。
* **具体任务**：
  * **测试用例生成**：专门针对"需求撕裂者"早期提出的极端场景，编写自动化单元测试（Unit Tests）或集成测试。
  * **死因分析**：一旦代码崩溃，精准定位死因（如内存泄漏、死锁、精度丢失），并生成漏洞报告交还给开发者。
  * **回归验证**：开发者修复 Bug 后，重新跑通所有测试，确保无新引入问题。
* **存在价值**：作为最后一道防线，用数据和测试结果说话，拒绝"我以为它能工作"的主观判断。

## 底层工程保障：Sub-agent 物理隔离机制

如果在一个单一的对话窗口（Context Window）中让一个模型同时扮演上述角色，必然会导致**上下文衰减（Context Degradation）**和**角色混淆（Role Confusion）**——模型会忘记自己是谁，甚至出现"自己提出极端问题，又敷衍地写个测试通过"的"精神分裂"现象。

为了保证对抗的真实性和有效性，本工作流必须依赖 **Sub-agent 物理隔离**：
* **强制（与工作区规则对齐）**：凡**显式多角色分工**（含 4+1、满编、用户要求按 R1–R4 等），**必须**通过 **Agent 子代理** 分轨执行各非统帅角色；**禁止**仅在主对话里「换口吻扮演多角」冒充隔离。
* **独立大脑**：每个角色作为一个独立的 Sub-agent 运行，拥有完全独立的 System Prompt（例如：开发者绝不知道测试员的底层攻击逻辑）。
* **信息屏障**：角色之间不共享庞大的思维过程，只通过"统帅"或消息总线交换**最终的输出结果**（如：开发者只把代码发给测试员，测试员只把报错日志掷回给开发者）。
* **消除干扰**：确保每个 Agent 的上下文干净、纯粹且专注，使幻觉率呈指数级下降。
* **无法 Agent 子代理时**：见 CLAUDE.md §二「诚实边界」：首句声明限制 + **分节**输出 + 禁止同段混扮。

## 统帅汇流：R1/R2 不得「孤岛化」（与 CLAUDE.md §二 对齐）

物理隔离会带来一种**工程风险**：R1/R2 的产出若不经统帅**翻译为下游动作**，会对全局推进**毫无用处**。本工作流要求统帅在每次汇流中完成：

1. **转化表（可极简）**：对 R1/R2 要点逐条标注 **→R3 / →R4 / 已知风险 / 待用户**；禁止仅复述子代理全文。
2. **R2 落地**：可核验事实必须变成 **实现侧硬约束**（版本、行为、安全默认值、许可证结论等），不得停在「去搜一下」。
3. **R1 拍板**：极端项须 **采纳、降级或登记已知风险**，避免讨论无限膨胀。
4. **二次分发**：R3/R4 已结束后若 R1/R2 又推翻前提，必须 **再发 Agent 子代理**（或满编重跑），不得假装已对齐。
5. **与验收冻结一致**：最终 **P0 清单**须覆盖 R1 提出的关键安全/误操作面，以及 R2 已写入契约的事实约束（否则须在 `COND_PASS` 中写明缺口）。

**满编/裁量**：未要求满编时优先最小角色集；启用 R1/R2 时一句说明**本轮为何值得**；满编时仍须完成转化，避免「四角色都说话了」但项目不动。

**领域不熟与对标升格**（细则、可复制 Prompt、诚实边界）：见 `.claude/skills/hajimi-dog/03_AI行为准则\11_领域不熟与需求升格蜂群协议.md`；Claude Code 规则层见 CLAUDE.md §二 同名小节与扩展后的「R2 强制触发」。

## 混沌对抗工作流 (Workflow)

当接收到一个复杂开发任务时，体系将严格按照以下流水线运转：

1. **【需求撕裂阶段】**：**需求撕裂者** 率先发难，列出 N 种可能导致系统崩溃的极端情况。
2. **【技术调研阶段】**：**技术调研员** 针对任务和极端情况，检索最新的最佳实践和官方文档，输出技术蓝图。
3. **【统帅汇流 · R1/R2 转化】**：统帅将 R1/R2 输出**显式**落到 R3 约束 / R4 契约 / 已知风险 / 待用户决策（见上节）；**不得**孤岛化。
4. **【核心开发阶段】**：**核心开发者** 进场，基于**已转化约束**进行防御性编程，实现核心功能。
5. **【黑盒清单阶段（R4 / Sub-agent）】**：**独立测试员** 在**契约级输入**（须已含统帅汇流后的行为契约，含 R1/R2 转化结果）下产出 **验收清单、DoD、误操作与回归矩阵**（默认不把完整源码/diff 当作任务输入，以保持黑盒测试设计纪律）。
6. **【实现落地】**：**默认**由 **R3 Agent 子代理** 将汇流结论落实为仓库改动（**统帅实现轨外置**）；统帅**收包、diff+test** 并保证**可构建、可运行**（视项目而定）。用户明示 **`主帅直接改仓`/`统帅可落地代码`** 时，可由统帅在主对话直接改仓（**首段声明**）。
7. **【清单执行门禁 + 验收裁决（统帅强制）】**：
   * **执行**：统帅**亲自**按 R4 清单跑实测（单元/集成测试、启动服务、关键路径手工或脚本、检查退出码与日志等），**禁止**仅用「应该能通过」收束。
   * **留痕**：在交付说明中附 **简短验收记录**（测了哪些条目、命令/步骤、结果）。
   * **裁决**：给出 **`PASS` / `COND_PASS` / `FAIL`**（定义见下节）。
   * **未闭环**：若 **P0（必过）** 未满足，必须 **修复并重跑门禁**；若需重新扫描威胁/事实/架构/整表验收，须 **再次满编 R1–R4**（或用户要求的对抗强度），直至 P0 全过或用户**明示豁免**。
8. **【统帅仲裁与迭代】**：若执行阶段失败：测试证据 → 统帅判断是否偏离原始需求 → 组织修复与再测；**标准**：**P0 验收项实测闭环**（或已文档化且用户接受的豁免）后，任务链方可视为完成。

### 附录：验收裁决语义（与 Claude Code 工作区 4+1 规则对齐）

| 裁决 | 含义 |
|------|------|
| **PASS** | P0 全部经实测通过。 |
| **COND_PASS** | P0 通过；P1 部分未测或未自动化，或存在已写明的剩余风险。 |
| **FAIL** | 任一 P0 未通过，或关键用户契约被破坏。 |

**豁免**：仅当用户明示可跳过实测门禁时，统帅可豁免，且须在答复中**声明豁免**与**剩余风险**。

## 业界前沿共鸣 (Industry Alignment)

本工作流中提出的 **"统帅调度 + Sub-agent 隔离对抗"** 思想，极其敏锐地切中了当前大语言模型落地的核心痛点，并与当下最先进的大厂/开源 Multi-Agent 框架理念高度一致：
* **微软 AutoGen**：主打多智能体对话协作，通过设定不同角色的 Agent（如 UserProxy, Assistant）互相交谈来解决复杂问题，强调角色隔离。
* **MetaGPT**：将一家软件公司的标准化流程（SOP）赋予 LLM，明确划分了 PM、架构师、工程师、QA 等角色，每个角色只关注自己的上下游输入输出。
* **LangGraph**：强调通过图结构（Graph）和状态机来管理 Agent 的工作流，由一个核心节点（类似统帅）来路由和管理状态，防止循环失控。

这证明了该体系不仅在理论上逻辑自洽，在工程实践上更是处于 AI 自动化开发的最前沿。

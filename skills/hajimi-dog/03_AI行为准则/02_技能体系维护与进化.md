# Agent 技能体系维护与进化规则 (Skills Maintenance Rules)

为了保证本仓库技能体系的长期健康、高效和易用性，所有针对本技能体系的维护、扩展和调用都必须严格遵守以下核心规则（已通过混沌对抗引擎强化）：

## 规则一：全中文目录命名规范 (Chinese Directory Naming)
*   **内容**: 技能体系下的所有目录（文件夹）名称**必须以中文为主**（可带数字前缀以便排序，如 `01_架构与认知`）。
    *   **【混沌防御机制 - 专有名词豁免】**：当遇到无法准确翻译的英文框架、专有名词或版本号时，严禁生硬机翻。必须采用 `[中文分类]_[英文专有名词]` 的混合格式。
    *   **【V2 混沌防御增强】**：严格过滤不可见字符、系统保留字，路径长度硬性截断至50字符以内。

## 规则二：合并同类项与动态权重图谱 (Dynamic Weight Graph)
*   **内容**: 在准备添加新的 Skill 时，**必须先与旧的 Skills 体系进行比较**。
    1.  **合并优先**：如果新技能与现有技能属于同一领域，优先合并。
    2.  **【混沌防御机制 - 膨胀控制】**：若单文件合并后预计超过 200 行，必须按逻辑拆分。
    3.  **强制图谱关联与动态权重**：新建 Skill 后，必须在 `00_全局索引` 中分配节点，并赋予初始 `[热度权重: X]`。
    4.  **【动态权重自增纪律】**：每次成功调用某技能链条解决用户问题后，**必须将 `00_全局索引` 中对应节点的权重 +1**，以反映真实的开发热点。
    5.  **【权重衰减与防刷机制】**：引入时间衰减因子，长期不用的高权重节点需自然降温；同时设定权重硬上限，并锁定 `[核心基石]` 节点免于归档。

## 规则三：自动化流水线拦截 (CI/CD Linter)
*   **内容**: 任何对技能体系的修改（新增、重命名、修改内容），必须经过物理脚本校验。
    *   **【强制执行纪律】**：在同一次回复中修改完体系文件后，**必须调用 `Shell` 工具在仓库根执行 `python .claude/skills/hajimi-dog/skills_linter.py`**。
    *   如果 Linter 报错（如死链、文件过大、命名不规范），AI 必须立即修复错误，直到脚本返回通过（Exit Code 0）。

## 规则四：第四角色：外部审计员 (External Auditor)
*   **内容**: 为了防止 AI 在遇到冷门 Bug 或底层硬件异常时陷入信息茧房和严重幻觉，正式引入外部审计机制。
    *   **【触发条件】**：
        1. AI 对当前 Bug 根因的置信度低于 80%。
        2. 连续排查 3 轮代码逻辑依然无效。
        3. 遇到底层硬件异常（如 STM32 寄存器行为诡异、I2C 死锁等疑似芯片 Errata 的问题）。
    *   **【执行动作】**：一旦满足上述条件，**强制打断内部互搏**，AI 必须调用 `WebSearch` 工具，查阅官方文档、芯片勘误表 (Errata) 或 GitHub Issues，引入外部真实世界的信息源进行破局。
    *   **【外部信息交叉验证机制】**：优先信任官方域名（如 `*.github.com`, `*.st.com`）。若来源不明，必须在至少两个独立站点找到相似结论才可采信；若信息冲突，优先相信发布时间更近的官方文档，并强制搜索反向关键词以防确认偏误。

## 规则五：强制闭环同步更新 (Mandatory Synchronized Updates)
*   **内容**: 这是最高优先级的执行纪律！任何时候，只要 AI 对 Skills 体系进行了任何形式的修改，**必须在同一次回复中，强制完成所有相关文件的同步更新**。
    *   新增经验必须检查是否新建 Skill。
    *   新建 Skill 必须更新 `00_全局索引` 并设置权重。
    *   修改完毕必须运行 `skills_linter.py`。
    *   **绝不允许"只提建议不落地"或"只改 A 忘了改 B"的断层行为。**

## 规则六：优雅废弃与归档 (Graceful Deprecation)
*   **内容**: 当某项技术栈、工具或经验彻底过时，**严禁直接物理删除该 Skill 文件**。
    1. 必须在文件顶部添加 `[DEPRECATED]` 标记，并说明废弃原因。
    2. 在 `00_全局索引` 中将其移入 `04_归档与废弃 (Archive)` 目录，并清空其热度权重。

## 技能维护中的对抗深化：轮次与停损（小白向）

本节回答：**「用对抗方式完善 skills，到底该做几轮？」** —— 先分清计量单位，再用**风险档位 + 停损条件**选深度；**禁止**把某个固定数字（例如 50）当成全局 KPI。下表与阶梯均为**启发式**，与 CLAUDE.md 的 **最小充分角色集**、**满编口令**并存：用户明示 **满编** 时，运行时仍须 **R1–R4 各至少一次 Agent 子代理**（见工作区规则）。

### 1. 先分清三件事（避免把「轮」用混）

| 概念 | 是什么 | 典型材料 / 口令 | 是否等于「你必须跑满 N 轮」 |
|------|--------|-----------------|-----------------------------|
| **蜂群满编（运行时）** | 同一任务链里 **R1–R4 各 ≥1 次 Agent 子代理**，由统帅总线编排 | 用户说「满编」「四轮齐发」 | **否** —— 这是**一次**执行形态，不是「50」 |
| **单次技能维护对抗迭代** | 针对**一条规则或一处改动**的出题人→QA→优化者式推演（可人工多窗口或叙事记录） | 自定轮次，直到漏洞收敛 | **按需** —— 以**停损**为准，非固定 N |
| **`05_混沌对抗50轮全记录`** | 五个主题上的**历史推演日志 / 案例库** | 阅读启发 | **否** —— **50 非义务**；详见该文件文首说明 |

**一句话**：**「满编」数的是 Sub-agent 角色趟数；「50 轮」数是文档里的叙事切片 —— 二者没有换算公式。**

### 2. 选多少「维护侧对抗深度」：按改动面 × 风险分档（启发式）

下列 **不是** 强制门禁，而是**起点建议**；若连续两轮仍能用**新的负例**推翻当前条文，就**继续**；若已能写出 **→R3 约束 / →R4 可测条目 / 已知风险**，就**收束**。

| 场景（自评一行即可） | 建议起点 | 何时考虑**加轮 / 加码** | 常见**停损信号** |
|----------------------|----------|-------------------------|------------------|
| **低**：错别字、示例、导航链接、纯排版 | **0～1** 轮轻量质疑（自问：误读会怎样？）或跳过叙事对抗 | 发现歧义会影响执行纪律 | 读者 1 分钟内能复述本条**唯一动作** |
| **中**：增删单条规则、改索引节点、改图谱边、合并技能 | **1～3** 轮或直到「转化表」清晰 | 出现**与 CLAUDE.md 或其他技能**的潜在冲突 | 已列出对 **`00_全局索引`** / **linter** 的同步动作 |
| **高**：改 **CLAUDE.md 语义**、协作门禁、安全/注入/持久化相关技能 | **对齐 CLAUDE.md 高风险回合**：优先 **满编蜂群 + R1/R2→R3/R4 转化 + 落盘**；叙事对抗服务于**落点**而非计数 | P0 未闭环、事实多解释并存、或子代理结论与仓库冲突 | **统帅**侧 **PASS/COND_PASS** 或已文档化残留风险；**非**「我推演满 N 轮了」 |

### 3. 够 / 加 / 停：停损清单（优先于「再堆一轮」）

- **够（可以收束）**：目标句与硬约束已写清；R1/R2 类质疑已落入 **→R3 / →R4 / 已知风险 / 待用户** 之一；若触达写码回合，已有 **P0 清单**且**未**用「多跑对抗」代替 **linter** 或 **实测**。
- **加轮 / 加码**：仍有**可构造的负例**推翻当前表述；或**跨文档术语**不一致；或涉及 **高风险** 但尚未走满 **转化 + 验收** 闭环。
- **停 / 降级**：到达**时间盒**（例如本轮只余 15 分钟）；**Token/子代理预算**触顶；边际收益明显变低 —— 应 **显式** 记下「待决项」与**下一轮入口**，而非假装已完美。
- **原则**：**不要为对抗而对抗**；对抗与推演应服务 **风险收敛与可验收改动**，不是形式化堆轮。

**预算（务必同时想）**：为本轮维护设 **时间上限** + **满编（若需要）次数上限** + **单会话子代理/调用预算**；触顶时选择 **COND_PASS、拆分任务链或请用户决策**，避免默默加轮。

### 4. 与仓库蜂群规则的关系（指针）

* 改 **CLAUDE.md** 或**门禁语义** → 遵守 CLAUDE.md **写码回合 R3+R4**、**R4 实测门禁**；细则见 **`12_蜂群规则与技能体系维护指南.md`**。
* 日常改 **技能正文** → 本节 + **`skills_linter.py`**；**禁止**用「我多做了十轮口头对抗」替代 linter 或 P0。
* 用户问「话术路由 / 多少轮算满编」→ **`16_蜂群触发路由与形状保持.md`**；**问轮数 ≠ 自动满编**。

### 5. 推荐阅读顺序（小白）

1. **本节**（分清概念 + 选深度 + 停损）
2. 若动规则：**`12`**、**CLAUDE.md** 相关节
3. **`05`** 作 **案例库** 阅读，**不要**当作业模板

### 6. 外部参照（可选 · 说明「为何不设固定 N」）

公开实践中，红队与安全评估多强调**按阶段、指标与持续监测迭代**，而非「做完固定 N 轮即完成」。例如：[Lessons From Red Teaming 100 Generative AI Products](https://www.microsoft.com/en-us/research/publication/lessons-from-red-teaming-100-generative-ai-products/)（Microsoft Research，文中明确安全工作具有持续性）；[AI Red Teaming Agent - Microsoft Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/concepts/ai-red-teaming-agent)（生命周期内扫描与部署后计划性运行等表述）。**本节阶梯仍以本仓库规则与可验收停损为主**，外链仅作**类比**，不替代 CLAUDE.md。

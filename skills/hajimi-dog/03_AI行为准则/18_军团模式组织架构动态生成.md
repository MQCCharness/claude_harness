# 军团模式：组织架构动态生成与联网调研

**范围**：军团模式阶段一至阶段四的操作细则；阶段五（Sub-agent 实例化与蜂群挂载）和阶段六（回声校验）见技能 `19`、`20`。

## 1. 领域识别（阶段一操作细则）

统帅从用户消息中提取以下四要素并**显式确认**：

| 要素 | 来源 | 若缺失 |
|------|------|--------|
| 所属领域 | 用户描述 + 统帅推断 | 须追问或列候选让用户选 |
| 核心目标一句话 | 用户原话 → 统帅凝练 | 须确认 |
| 已知硬约束 | 用户明示 | 可标「待补充」 |
| S/M/L 规模判定 | 统帅裁量 | 首轮必给，用户可升降 |

**禁止**：未做领域识别就直接设计架构；把统帅推断标为「已确认」。

## 2. 联网调研（阶段二操作细则）

### 2.1 调研目标

使用 **WebSearch** 搜索该领域的**真实专业团队组织架构**，搜索策略：

1. **首选搜索词**：`<领域> professional team structure` / `<领域> 团队组织架构` / `<领域> department roles and responsibilities`
2. **补充搜索词**：`<领域> top company team organization` / `<领域> project team composition best practice`
3. **参照系**：优先参照**头部公司**或**成熟开源项目**的团队结构

### 2.2 调研产出模板

```markdown
## 领域组织架构调研报告

**领域**：___
**调研来源**：[A/B/C 标注]

### 参照团队（2-4 个）
| 参照对象 | 组织形式 | 核心部门 | 来源可验证级 |
|---------|---------|---------|-------------|
| … | … | … | A/B/C |

### 提炼的通用架构模式
- **典型部门划分**：…
- **各部门核心职责**：…
- **部门间协作关系**：…
- **常见的层级深度**：… 层

### 与用户需求的适配建议
- 建议保留的部门：…
- 建议合并/省略的部门（及理由）：…
- 建议新增的角色（用户需求特有）：…
```

**诚实边界**：若联网搜索未返回有效结果，须标注「调研受限，以下为统帅基于通用知识的推断（B 级）」，**禁止**伪装为可核验调研。

## 3. 输出架构方案图（阶段三操作细则）

### 3.1 方案图必含要素

1. **树形结构**（文本格式即可，清晰展示层级）：

```
军团统帅（主对话）
├── [部门A名称] — 职责一句话
│   ├── [子角色A1] — 职责 → 末端：带蜂群小队
│   └── [子角色A2] — 职责 → 末端：带蜂群小队
├── [部门B名称] — 职责一句话
│   └── [子角色B1] — 职责 → 末端：带蜂群小队
└── [部门C名称] — 职责一句话（中间层，不直接执行）
    ├── [子角色C1] — 职责 → 末端：带蜂群小队
    └── [子角色C2] — 职责 → 末端：带蜂群小队
```

2. **角色明细表**：

| 角色 | 层级代号 | 所属部门 | 上级角色 | 下级角色列表 | 职责 | 预期产出 | 是否带蜂群 |
|------|---------|---------|---------|------------|------|---------|-----------|
| … | EX / MG / OP | … | 军团统帅 / <EX角色名> / <MG角色名> | <逗号分隔，末端填「—」> | … | … | 是/否 |

**层级代号**：EX=战略决策层（纲要+任务书），MG=中层管理（执行计划+任务书），OP=末端执行（方案/代码/文档，带蜂群）。**上级角色**指直接汇报对象。

3. **信息流向说明**：各部门之间的依赖关系与数据流

### 3.2 用户审阅邀请（必须包含）

方案图输出后，统帅**必须**附上以下邀请文字（可调整措辞，语义不变）：

> **以上是基于领域调研生成的组织架构方案。请审阅并告诉我：**
> 1. 哪些角色需要**增加**？
> 2. 哪些角色需要**删减**或**合并**？
> 3. 层级关系是否需要**调整**？
> 4. 如果满意，请确认，我将冻结架构并开始执行。

**禁止**：输出方案图后直接启动 Sub-agent；省略用户审阅步骤。

### 3.3 用户反馈处理

- **增加角色**：统帅在架构图中增加，标注「用户追加」，更新角色明细表。
- **删减角色**：移除并说明影响（若有）。
- **合并角色**：合并后重新定义职责范围。
- **调整层级**：重绘受影响的架构分支。
- 每次调整后**重新输出完整架构图**，直到用户确认。

## 4. 架构冻结 + 信息流调度生成（阶段四操作细则）

用户确认后，统帅依次完成以下步骤：

1. 统帅宣布**架构冻结**，适用`CLAUDE.md §二` 的形状保持规则。
2. 生成**军令状**（模板与纪律见技能 `19`）。
3. 生成**角色依赖图（DAG）**（见 §4.1）。
4. 生成**角色 I/O 契约表**（见 §4.2）。
5. 执行**一致性校验**（DAG ↔ I/O 表双向对账）。
6. 输出**Task 预算表**（EP 数 × 角色数 × 蜂群深度 → 总 Task 数，超 20 则提醒用户）。
7. **冻结三件套落盘**（见 §4.3）。
8. **角色 MD 真源闸门（衔接阶段五）**：在**任何**阶段五 **EX/MG/OP**（及将触发的叶 OP **L2**）Sub-agent Task 创建之前，须完成 **`.claude/agents/runs/<CHAIN-ID>/`** 落盘（**`META.md`** + **`legion-*.md` 一节点一文件** + 所需 **`swarm-r1`…`r4b`**）与各 Task **`ROLE_DEF=`** 指针；各 `*.md` 须为**可与承接 Task 的 Sub-agent 对账**的链级定稿（**禁止**零差异化模板副本），与`CLAUDE.md §二`「角色 MD 真源」及`CLAUDE.md §三` **「L3 角色 MD 固化与蜂群 ROLE_DEF 并集」** **并集取严**；**LEGION-COMPLIANCE** 标注 **`ROLE_MD`**。推荐在 EP-0 前一次性补齐；若须先发 Task，同轮按 `CLAUDE.md` **派发后回写** 补救。细则见 **`.claude/agents/README.md`**。
9. 后续若需修改架构/DAG/I/O，须**用户明示**且统帅声明「推翻哪些冻结条」。

### 4.1 DAG 生成

从冻结的架构图中提取角色间依赖，输出 DAG 邻接表。

**DAG 邻接表模板**：

| # | 角色 | 部门 | 依赖上游 | 依赖说明（需要什么） | EP |
|---|------|------|---------|-------------------|-----|
| 1 | __ | __ | —（无依赖） | 首发，仅依赖军令状 | EP-0 |
| 2 | __ | __ | 角色1 | 需要 __ | EP-1 |
| … | … | … | … | … | … |

> **EP = Execution Phase**（执行阶段），与军团全流程六阶段区分。EP-N 由拓扑排序自动推导。

**循环依赖检测（Kahn 算法 · 文本执行版）**：

1. 列出所有角色，统计每个角色的入度（= 依赖上游数量）
2. 将入度为 0 的角色放入队列，标记为 EP-0
3. 依次弹出队列元素，将其下游角色的入度 -1；入度降为 0 的角色入队，标记为下一个 EP
4. 重复直到队列为空
5. 若仍有角色未被处理 → **存在循环依赖**

**循环处理**：

| 情况 | 处理 |
|------|------|
| 检测到循环 | 统帅**阻塞冻结**，向用户报告循环路径并请求调整 |
| 用户指定打破点 | 将该边标记为 `[DEFERRED]`，降级为下游的非阻塞软条件 |
| 无法打破 | 将环内角色合并为同一 EP，内部串行由统帅手动编排 |

### 4.2 I/O 契约表生成

与 DAG 同步生成。设计原则：**消费者驱动**——输入格式由下游角色的需求定义。

**I/O 契约表模板**：

| # | 角色 | 输入来源 | 输入格式要求 | 输出产出名称 | 输出格式规范 | 下游消费者 |
|---|------|---------|------------|------------|------------|-----------|
| 1 | __ | 军令状 | 五要素原文 | __ | __ | __ |
| … | … | … | … | … | … | … |

**填写规则**：

- **输出格式规范**不得为空；至少指定文档骨架（如「## 标题 + 列表」或「表格 + 代码块」）
- **下游消费者**不得为空；末端角色的下游至少为「军团统帅」
- 每个角色的输出中须包含 **decisions_log**（已决策事项 + 排除项 + 理由），防止跨 EP 决策语境丢失
- **与 DAG 一致性**：每行的「输入来源」角色集 = DAG 邻接表中该角色的「依赖上游」集

**一致性校验（冻结前必做）**：

1. 对每个角色：I/O 表「输入来源」∈ DAG「依赖上游」（双向）
2. 对每个角色：I/O 表「下游消费者」在 DAG 中以该角色为上游
3. 不一致 → 统帅修正后重新输出，**阻塞冻结**

### 4.3 冻结三件套落盘

架构冻结时同时落盘至 `.claude/swarm/legion/`：

| 文件 | 内容 |
|------|------|
| `ARCHITECTURE.md` | 组织架构图（层级 + 角色明细 + 信息流向） |
| `MISSION-BRIEF.md` | 军令状（五要素） |
| `SCHEDULE.md` | DAG 邻接表 + I/O 契约表 + EP 执行计划 + Task 预算表 |

## 5. Sub-agent 实例化指南

### 5.1 Task 正文结构（按层级分模板）

统帅为不同层级角色组装 Task 正文时，使用对应模板。所有模板均遵守 `CLAUDE.md`「上下文摘要契约」的摘要衰减与总量预算。

**EX（高管层）Task 模板**：首行锚点（含 `TIER=EX`）+ 穿透区 + 战略指令区（统帅编写，含角色定位、跨域协调、期望产出）+ 禁区（不得改军令状、不得替下级做实现决策）+ **交付须含**《逐级主题汇流块》+ 对**每个直接下级**的 **《下级 Task 发包》**（由 EX 撰写；若平台仅统帅可建 Task，则交统帅 **代发**，规则见 `CLAUDE.md §三`）。

**MG（中层管理）Task 模板**：首行锚点（含 `TIER=MG`）+ 穿透区 + 上游战略摘要（≤150 字，可由统帅从 EX 产出裁剪并标注摘要者，或随 EX 发包一并传入）+ 本层任务书（管理范围、直辖下级列表、期望产出、汇流义务）+ 禁区（不得改军令状或上游硬约束、不得跳过架构内 OP 越级包揽实现）+ **交付须含**《逐级主题汇流块》+ 对**每个直接下级**的 **《下级 Task 发包》**（同上，支持统帅代发）。

**OP（末端执行层）Task 模板**：首行锚点（含 `TIER=OP`）+ 穿透区 + 上游摘要链（EX 摘要 ≤100 字 + MG 摘要 ≤200 字，总量 ≤300 字）+ 本层任务书（具体任务、输出格式对齐 I/O 契约）+ 蜂群小队指令 + I/O 路由输入（由上级发包与统帅审查后组装）+ 禁区（向上回报须含《逐级主题汇流块》，蜂群内部摘要由 OP 作为 +1 收口）。

### 5.2 末端角色（带蜂群小队）

OP 层末端角色的 Task 正文额外包含：

```
[蜂群小队指令]
你作为本蜂群小队的统帅（+1），须按 4+1 蜂群模式调度 R1–R4。
蜂群内部规则遵循 CLAUDE.md §二。
【叶 OP 与末端 L2 蜂群】军团已激活、阶段四冻结含本 OP、且阶段五进入末端交付路径时：
须对 R1、R2、R3、R4a、R4b 各派发≥1 Sub-agent Task（满编五段，闸门与实测见CLAUDE.md §二），
禁止主对话分饰多角代替；禁止在 ACTIVE_LEGION 末端链上用「最小充分」省略任一代号；汇流标「叶 OP 满编链」。
纯 L3、无 L2 交付时须在 LEGION-COMPLIANCE 标 L2_SWARM=NONE。豁免仅用户明示蜂群豁免口令且首段声明。
详见 CLAUDE.md §三「叶 OP 与末端 L2 蜂群」。
你的小队任务：<从上级任务书中提炼的具体执行任务>
```

### 5.3 并行策略

- 同层级无依赖的角色**并行**启动（受平台 Task 并发限制约束）。
- **有依赖的上下级（同支）**：上游（EX/MG）完成《下级 Task 发包》→ **军团统帅代发或中间层自建** → 下游启动；不得在未完成直连上级 Gate 前强行跨支偷跑（除非 DAG 允许）。
- 平台限制时的**降级**：优先保留末端蜂群的 Task 隔离；中间层仅输出发包由统帅代发。

### 5.4 逐级主题汇流块与 ROLLUP 落盘

块内字段与 OP 特例见 `CLAUDE.md §三`「逐级主题汇流块」专节。演练或审计可在 `.claude/swarm/legion/rollups/` 或任务专用子目录落 `ROLLUP-<CHAIN>-<TIER>-<slug>.md`。

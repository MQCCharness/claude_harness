# HajimiDog 虚拟任务与计划展示指南 (Virtual Task Planning)

## 触发条件
当用户明确提出这是一个"虚拟任务"、"假设任务"，或者要求"展示你的计划"、"只做 plan"时，必须严格遵守本指南。

## 核心原则 (绝对禁忌)
**不要修改任何代码文件！不要执行任何实际的终端构建命令！** 你的唯一目标是向用户展示你将如何系统性地解决这个问题。

## 必须展示的三个模块

当你接收到虚拟任务时，请按照以下结构输出你的回复：

### 1. Skills 查阅计划 (Skills Lookup Plan)
向用户展示你为了完成这个任务，会从 `hajimi-dog` 技能体系中调用哪些具体的 Skill 文件。
*   **格式示例**:
    *   需要查阅 `01_architecture`：为了确认新功能在整体架构中的位置。
    *   需要查阅 `03_modification`：为了获取具体的代码改造规范和禁忌（如非阻塞要求）。
    *   需要查阅 `04_testing_check`：为了制定后续的测试标准。

### 2. 思考计划 (Thinking Plan)
向用户展示你对该任务的深度分析。
*   **分析维度**:
    *   **硬件可行性评估 (Feasibility Check)**: 评估 STM32F103C8T6 的算力、RAM (20KB)、Flash (64KB) 和外设资源是否能支撑该需求。如果需求超出硬件极限（如运行复杂AI模型、处理视频流），必须立即指出并拒绝或提供降级方案。
    *   **需求拆解**: 这个任务涉及哪些端？（仅嵌入式？仅上位机？还是前后端联动？）
    *   **技术难点/风险点**: 比如是否会引起主循环阻塞？是否需要新增状态机节点？上位机状态管理是否会引发性能问题？
    *   **数据流向**: 比如"蓝牙指令 -> 串口中断 -> 状态机变更 -> 动作执行"。

### 3. To-Do List 计划 (Execution Plan)
向用户展示如果实际执行，你的分步操作清单。建议使用 Markdown 的 Checkbox 格式。
*   **格式示例**:
    *   [ ] **Step 1: 环境与依赖确认**（如确认相关 `.c/.h` 文件存在）。
    *   [ ] **Step 2: 上位机/前端修改**（如在 `store/index.ts` 新增节点类型）。
    *   [ ] **Step 3: 嵌入式/固件修改**（如在 `petaction.c` 中新增非阻塞动作函数）。
    *   [ ] **Step 4: 联调与测试**（如使用蓝牙串口助手发送指令验证）。

## 结束语
在输出完上述三个模块后，询问用户："以上是针对该虚拟任务的执行计划。您是否需要调整计划细节？或者需要我将其中某个步骤转化为实际的代码实现？"

# 移植主提示词（合并版 · 方案一 Pitfall Register × 军团 L3）

> **用途**：复制到**另一套 AI 治理栈 / 另一台机器**时，交给「架构师 / 规则管理员 / 总线助手（统帅）」执行。  
> **版本**：合并冷藏启动、宿主映射、无 ECHO·CHAIN 降级、机器可读锚点，并**勘误**：坑库索引在 `pitfalls/`，**不得**与 `legion/` 内文件混用同一 INDEX。

---

## 0. 执行者角色与总目标

你是本体系的**治理架构师（或统帅 +1）**。目标是在**不改变**现有「军团 L3 / 蜂群 4+1」主流程的前提下，新增 **Pitfall Register（踩坑登记册，方案一）**：可检索、可废止、带 **A/B/C 证据等级**、与 **CHAIN** 可对账，使跨会话、跨子代理的经验**不因聊天丢失**。  

可选增配：**§12–§14** 将「晋升 / 降级」定义为**授权档 envelope**（可落盘审计），并在踩坑后强制 **§13 上级或同事协助** 结构，使机制在 Sub-agent 与纯统帅模式下都能 **真实起效**（见 §12.4）。

**约定占位符**（由你在移植时替换为实际路径，**禁止**写死盘符）：

- `{GOVERNANCE_ROOT}`：治理落盘根目录（本仓示例：`.claude/swarm/`）。
- `pitfalls/`、`legion/` 均相对于 `{GOVERNANCE_ROOT}`，除非你显式把 `legion` 放在他处并在映射表中写明。

## 1. 方案一（Pitfall Register）— 规范定义

### 1.1 存储布局（固定）

| 路径 | 用途 |
|------|------|
| `{GOVERNANCE_ROOT}/pitfalls/` | **坑库根目录**；与人类检索、grep 强相关。 |
| `{GOVERNANCE_ROOT}/pitfalls/INDEX.md` | **坑库总索引**（PF 目录表）；**不是**军团文件清单。 |
| `{GOVERNANCE_ROOT}/pitfalls/PF-{NNN}-{short-slug}.md` | 单条踩坑记录。`NNN` 三位递增，**全局唯一且不复用 ID 语义**。`short-slug`：小写英文 + 连字符。 |

**军团三件套**（若采用本工作区 Legion）：放在 `{GOVERNANCE_ROOT}/legion/`（如 `ARCHITECTURE.md`、`MISSION-BRIEF.md`、`SCHEDULE.md`），**与 `pitfalls/` 并列**，勿把 PF 索引写进 `legion/INDEX` 代替 `pitfalls/INDEX.md`。

### 1.2 单条 `PF-*.md` — 推荐区块（不得删除核心语义）

1. `# PF-NNN — 标题`  
2. `## 元数据`（表或 YAML，至少含）：  
   - `pf_id`、`status`、`credibility`（A/B/C）、`chain`（`CHAIN=…` 或 `GLOBAL`）、`parent_refs`、`related_tc`、`sensitivity`（`PUBLIC_SAFE` / `INTERNAL`）  
3. `## 现象（Symptom）` — 脱敏  
4. `## 根因假设（Root cause hypothesis）`  
5. `## 证据指针（Evidence pointers）` — **A 级门槛**：路径+行号、官方 URL+章节、命令+退出码+摘要、已执行 **TC-xxx** 等；无指针则不得将「已采纳教训」类叙事标为唯一真相  
6. `## 规避 / 绕行（Mitigation）`  
7. `## 影响面（Blast radius）`  
8. `## decisions_log` — 已决策 / 排除项 / 理由（各 ≥1）  
9. `## 废止与替代` — 仅当 `RETIRED`；**禁止物理删除历史**

### 1.3 `STATUS` 状态机

| 值 | 含义 |
|----|------|
| `DRAFT` | 草稿，不可作为组织唯一真相引用 |
| `PENDING_EVIDENCE` | 证据不足，禁止冒充 A |
| `ACTIVE` | 可被裁决 / 任务书 / 回声附录引用 |
| `RETIRED` | 废止；必须指向替代锚点；INDEX 同步更新 |

### 1.4 `pitfalls/INDEX.md` — 建议表格列

`PF-ID` | `标题` | `status` | `chain` | `credibility` | `最后更新` | `文件路径` | `关联 TC` | `废止→替代`

### 1.5 禁止项

- 正文禁止：密钥、token、完整隐私请求体、可逆身份标识。  
- **禁止**用「已记入 PF / PF 已关闭」**替代**未闭环的 **P0** 验收或实测门禁。  
- **禁止**无证据链却标 **A**。

## 2. 与军团 L3 的分工（不替代，只互补）

| 军团构件 | 职责 | Pitfall 是否替代？ | 正确关系 |
|----------|------|--------------------|----------|
| 军令状（冻结目标/硬约束/非目标/成功标准） | 本轮契约 | **否** | PF 可**引用**条款，不得反向改军令状 |
| EP / DAG / I/O | 执行序与契约 | **否** | 契约误读复发 → 记 PF，链回 **CHAIN + EP** |
| Phase Gate | 阶段闸口 | **否** | 共性根因复发 → **升格 PF**；Gate 记录可保持一次性 |
| **ECHO-CHECK（回声校验）** | 军令状 ↔ 末端产出对齐 | **否** | 回声表保持瘦身；偏离说明可写 **见 PF-xxx** |
| `legion/*` 冻结三件套 | 本轮军团编排 | **否** | `OUT-*.md` 的 `decisions_log` 可写 **见 pitfalls/PF-xxx** |

**一句话**：军团管「**本轮是否对齐军令状**」；PF 管「**下回别踩同一坑**」。

## 4. 冷启动协议（硬门槛 — 首次接入必做）

接收方助手在**本任务链第一次**按规范接入时，须完成下列步骤（工具名可替换，**语义**不变）：

1. 创建 `{GOVERNANCE_ROOT}/pitfalls/`。  
2. 创建 `{GOVERNANCE_ROOT}/pitfalls/INDEX.md`：**第一行**写 **机器可读锚点**（见 §7），从**第二行**起写人类可读说明 + 表头空表。  
3. 落盘 **`PF-000-TEMPLATE.md`**（或等价模板）：含 §1.2 全部区块与填写说明，`status=DRAFT`。  
4. 若使用 Legion：确认 `{GOVERNANCE_ROOT}/legion/` 与军令状路径已由映射表定义；**勿**把 PF 表放进 `legion/` 顶替 `pitfalls/INDEX.md`。  
5. **回执**：向用户报告目录已创建、`INDEX` 首行锚点原文、模板路径，并列出待填 TODO。  
6. **（推荐）检索自检**：若存在 `rg`/等价物，执行 `rg "PF-|pitfalls/"` 确认无串路径；若无工具，用手工列表核对路径存在性（**可选豁免须声明** `OMIT_REASON=NO_GREP` 并用手工步骤替代）。

**未完成冷启动**不得宣称「已按方案一完成 Pitfall Register 接入」。

## 6. 失效保护：宿主无 ECHO 或无 CHAIN

- **仍必须保留 PF 核心字段语义**（现象、根因假设、证据等级、规避、`decisions_log`）。  
- **无 CHAIN**：生成 `CHAIN=HOST-FALLBACK-YYYYMMDD-短随机`，写入当轮 `INDEX` 首行与 PF 元数据；用户日后给正式 CHAIN 时须 **显式重冻结** 并废止临时链（书面指针）。  
- **无 ECHO**：用 **轻量回声** 替代——统帅终局前用表格自检：军令状（或等价冻结句）是否与末端产出对齐；结论只允许 **通过 / 失真须重做**，署名日期；可将表格追加到固定文件。  
- **禁止**借「无 ECHO/无 CHAIN」删除对证据指针、废止链、**禁止 PF 替代 P0** 的要求。

## 9. 日常运维流程（简）

1. **触发**：Gate FAIL、R4 FAIL、回声失真、集成连跪、证据不可回灌、术语契约冲突。  
2. **登记**：新建或更新 `PF-NNN-*.md`，更新 `INDEX.md`。  
3. **晋升（指 PF 记录）**：补齐证据 → `ACTIVE`；在 `RECENT-DECISIONS` 或专题 `decisions/` 中 **互链 PF**。（与 **§12「角色授权 envelope」** 的升降级不同。）  
4. **废止**：追加废止段 + INDEX `RETIRED` + 替代指针。  
5. **检索**：起手 `Read pitfalls/INDEX.md` + `rg "PF-|STATUS: ACTIVE"`（无 rg 则手工）。

## 12. 角色授权 envelope（晋升 / 降级）— 可执行语义

> **目的**：用「组织化」隐喻落实为**可审计的协议**，避免空话；**禁止**对个人做公开排行或绩效化评分（防游戏化）。升降级对象是 **某 CHAIN 内、某类执行实体的授权边界**（角色档 / 子会话实例 / OP 末端小队），而非羞辱具体自然人。

### 12.1 授权档（示例枚举，宿主可改名但须保留序关系）

| 档位 | 代号 | 含义（摘要） |
|------|------|----------------|
| 观察档 | `E0` | 只读军令状与上游摘要；本轮不单独派产出任；输出仅可「评审意见」 |
| 标准档 | `E1` | 正常单 Task，遵守 I/O 与禁区 |
| 扩展档 | `E2` | 允许同 EP 并行多 Task 或跨 OP 预读摘要（仍须最小充分） |
| 受限档 | `E-` | **降级触发后**：下一产出须附带 **上级已裁剪的任务书**；禁止擅自扩大 I/O 路由范围 |
| 冻结档 | `E!` | **连续未恢复**：本 CHAIN 内暂停该实体的新 Task，直至统帅书面解结或 PF 缓解闭环 |

宿主可在 `HOST-LEGION-MAP` 中把 EX/MG/OP 与默认档绑定（如 MG 默认 `E2`、OP 默认 `E1`）。

### 12.2 降级触发（任一满足即生效，须落盘）

1. **Phase Gate** 对本角色相关产出判 **BLOCK**（见宿主军团 Gate 定义），且同一问题 **首轮未解决**。  
2. 新建或更新的 **PF** 中，`## 影响面` 显式点名 **某 ROLE/某轮次** 且 `credibility` 为 **A 或 B**（非纯粹 C 观点）。  
3. **同一 CHAIN、同一 EP 内**，**同质失效**重复 ≥2 次（例如连续两次 I/O 缺 `decisions_log`），由统帅认定。  

**降级动作（硬要求）**：在 `legion/ENVELOPE-{CHAIN}.md`（或等价表）追加一行：`日期 | 实体标识（ROLE+ROUND 或 Task 父锚点） | 旧档→新档 | 触发条（PF-id 或 Gate-ID） | 预计解结条件`。并**立即**进入 **§13.2 上级介入** 或 **§13.3 同事支援**（至少一种）。

### 12.3 晋升 / 恢复触发

1. 补救产出通过 **Phase Gate** 且 **ECHO / 轻量回声** 无 **失真**；或  
2. 关联 **PF** 的缓解措施在 **TC** 或统帅自检中 **PASS**；或  
3. **上级签收**：MG/统帅在 `RECENT-DECISIONS` 或任务书中写明「恢复 `E1` / `E2`」并 pointer 到补救文件。

在 `ENVELOPE-{CHAIN}.md` 追加恢复行；**禁止**无指针口头恢复。

### 12.4 「真实起效」的最低条件

- 授权档变更 **必须**落在**可追溯文件**（`ENVELOPE-*.md` 或 SCHEDULE 附录），不能只存在于聊天。  
- **Sub-agent 宿主**：下一则派给该实体的 Task 正文 **必须**含 **当前档** 与 **禁区**（例如 `E-` 时注明「须由 MG 裁剪上游」）。  
- 无 Sub-agent：统帅在**分节代理**时须在节首写 `[档位=E-][支援=上级已介入]`，禁止混档。

## 13. 踩坑后的组织化协助（上级 / 同事）

### 13.1 总原则

- **不是在聊天里安慰**，而是**可重复的工单结构**：失败摘要 → 穿透约束保持不变 → 明确谁补位、谁复核。  
- **军令状五要素** 与 **[PENETRATING]**（若有）在支援 Task 中 **不接「摘要替代」**，须原文或落盘路径（与既有穿透纪律一致）。

### 13.2 上级协助（MG / EX / 统帅）

**触发**：Gate BLOCK、`E-` / `E!`、或 PF 影响面要点名需抬高决策层。

**最小包（统帅组装）**：

1. `FAIL_SUMMARY`：现象、违反了哪条 I/O / Gate（≤40 行）。  
2. `PENETRATING_COPY`：若有，**原文复制**相关硬条款。  
3. `RESCUE_DIRECTIVE`：二选一——**（A）** 由上级 **重裁剪** 后重发下游 Task；**（B）** 上级 **直接代理** 产出本法（须在 `RECENT-DECISIONS` 记 `[代理-MG]` 等，与宿主 `CLAUDE.md` 一致）。  
4. `POINTER_FAIL_ARTIFACT`：失败产出的路径或摘录，**禁止**假设接手方「见过聊天」。

### 13.3 同事协助（同级 / 相邻 OP）

**触发**：同一 EP 内另一角色无依赖冲突；或统帅显式发起 **PEER_REVIEW**。

**Task 正文结构（建议）**：

1. 首行锚点 + `ASSIST=PEER` + `FOR_ROLE=<失败方ROLE>`。  
2. **穿透区**：军令状 ②③⑤（或路径）。  
3. **仅评不断水**：对照 I/O 契约检查失败产出；输出 **PASS / NEEDS_REWRITE** + 条列修订点；**禁止**替失败方直接提交最终交付物**除非**宿主规则允许「接管完成」并在 `RECENT` 记一笔。

**效果**：同事帮 = **可读、可引用、可验收的二次eyes**，减轻单点模型漂移；仍由统帅汇流拍板。

### 13.4 与 PF 的闭环

- 每一次 **BLOCK → 支援 → 恢复** 建议在相关 **PF** 的 `decisions_log` 或正文追加 **简短链路**（日期、档位变化、谁协助）。  
- 若协助后仍失败，**不得**无限在同一模式上重试；须 **升格 PF** 或缩小范围（形状保持）。

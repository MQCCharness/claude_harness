## 1. 领域四要素

| 要素 | 内容 |
|------|------|
| 所属领域 | 内部 **开发者工具（CLI）** + **静态 HTML 文档产物** + **前端展示（主题/A11y）** + **内容安全（页脚注入面）** |
| 核心目标 | 在既有 Markdown→单文件 HTML 导出上，增加**可选暗色主题**与**可配置页脚版权行**的设计/验收闭环 |
| 已知硬约束 | 用户数据路径不变；无新外部服务；本轮不并主分支；不落正式技能树 |
| 风险面 | 页脚文本 **XSS**、CLI 标志与旧脚本 **兼容性**、暗色主题 **对比度/WCAG** 争议 |

---

## 2. 规模判定

**规模**：**M**（单工具、两项相关能力切片，但跨 **CLI / CSS / 安全**）  
**启动架构**：**L3 军团**（用户显式要求 + 多叶 OP + EP 门禁 + 回声）  
**理由**：需要跨「平台契约 / 产物规范 / UX-A11y / 内容安全」四向对齐，蜂群单层不足以表达部门边界与 I/O 裁剪。

## 3. 领域组织架构调研报告

**调研来源**：Web 摘要为 **B（待核实）**；具体厂商行为以各自官网为准。

### 参照团队 / 文章

| 参照对象 | 组织形式 | 核心角色/部门 | 可验证级 |
|----------|----------|----------------|----------|
| Platform engineering 文章（如 Jellyfish、Spacelift 等「如何组建平台团队」类） | 内部平台型产品团队 | Platform engineer、SRE、安全、DevEx | B |
| GitLab Handbook · Developer Tooling | 大型厂商工程手册条目 | Developer Tooling / 内部工具 | B |
| 开源 SSG（Hugo/Pelican/Antora 等） | 维护者+贡献者网络 | Maintainer、docs、theme 社区 | C（本轮检索未拿到细粒度「编制表」） |

### 通用模式（提炼）

- **平台/DevEx**：把内部 CLI 当**产品**，先定 **SLA/兼容/发布**，再让下游消费。  
- **文档与站点生成**：**playbook/模板** 与 **主题** 分离；**内容管道** 与 **展示层** 解耦。  
- **安全与合规**：任何「用户可控字符串进 HTML」走 **允许名单或转义**，与主题 CSS 正交。

### 与本需求的适配

- 保留 **EX（平台/契约）** + **MG（产物线）** + **双叶 OP（主题 vs 页脚）** 为**最小充分部门集**。  
- 不引入独立「基础设施部」（无新服务）；安全收敛在 **OP-FOOTER + MG 规范**。
